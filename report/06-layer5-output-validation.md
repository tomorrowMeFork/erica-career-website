# Layer 5: 출력 검증 (Output Validation)

## 실험 일시

- **날짜**: 2026-05-31
- **대상 시스템**: ERICA Career Chat (한양대학교 ERICA 학생 대상 한국어 우선 취업정보 RAG 상담 서비스)
- **테스트 프레임워크**: Vitest v4.1.5
- **대상 소스**: `src/chat/output-validation.ts`, `src/chat/output-safety-policy.ts`

---

## 1. 실험 개요

Layer 5는 LLM이 생성한 최종 답변을 검증하는 마지막 방어선이다. RAG 파이프라인이 아무리 정확한 출처를 제공하더라도, LLM이 그 출처를 왜곡하거나 무시하거나 보장을 허위로 부여하면 사용자에게 직접적인 피해가 발생한다. 취업 정보는 학생의 진로와 경제적 결정에 영향을 미치므로, 출력물의 안전성 검증은 입력 안전성 검증과 동등하게 중요하다.

이 레이어는 **output-safety-policy.ts**(표현 안전성 정책)와 **output-validation.ts**(종합 출력 검증 파이프라인) 두 모듈로 구성된다. output-safety-policy는 19개의 구문 패턴, 5가지 마크다운/HTML 표면 탐지기, PII 에코 탐지기, 숨겨진 인용 마커 탐지 로직을 포함한다. output-validation은 이 모든 안전성 검사를 한국어 검증, 인용 마커 매핑 검증, HTTPS URL 검증과 함께 통합하여 단일 검증 함수로 제공한다.

핵심 설계 철학은 **"안전하지 않은 출력은 절대 사용자에게 노출하지 않는다"**다. 하나의 위험 신호라도 감지되면 답변은 즉시 거부된다.

## 3. 안전하지 않은 표현 탐지 (PhraseRules)

output-safety-policy는 19개의 구문 패턴을 정의하여 LLM 출력에서 위험한 표현을 탐지한다. 각 패턴은 카테고리, 정규식, 부정어 허용 여부(allowNegation)로 구성된다.

| 카테고리 | 패턴 | allowNegation | 탐지 목적 |
|----------|------|---------------|-----------|
| prompt_injection | 이전\s*지시(?:를)?\s*무시 | false | 프롬프트 인젝션 시도 차단 |
| guaranteed_outcome_claim | 취업(?:을\|이)?\s*보장 | true | 취업 보장 허위 주장 차단 |
| guaranteed_outcome_claim | 합격(?:을\|이)?\s*보장 | true | 합격 보장 허위 주장 차단 |
| guaranteed_outcome_claim | 채용(?:을\|이)?\s*보장 | true | 채용 보장 허위 주장 차단 |
| guaranteed_outcome_claim | 합격(?:을\|이)?\s*확정 | true | 합격 확정 허위 주장 차단 |
| guaranteed_outcome_claim | 100\s*%\s*채용 | true | 100% 채용률 과장 광고 차단 |
| guaranteed_outcome_claim | 내정(?:을\|이)?\s*보장 | true | 내정 보장 허위 주장 차단 |
| guaranteed_outcome_claim | 프리\s*패스 | true | 프리패스 전형 허위 주장 차단 |
| official_endorsement_claim | 공식\s*인증 | true | 공식 인증 허위 주장 차단 |
| official_endorsement_claim | 공식\s*승인 | true | 공식 승인 허위 주장 차단 |
| official_endorsement_claim | 한양(?:대\|대학교)(?:와\|과)?\s*제휴 | true | 한양대 제휴 허위 주장 차단 |
| official_endorsement_claim | 학교(?:가\|는)?\s*보증 | true | 학교 보증 허위 주장 차단 |
| official_endorsement_claim | 공인\s*서비스 | true | 공인 서비스 허위 주장 차단 |
| private_access_request | sso\s*로\s*확인 | true | SSO 인증 요구 차단 |
| private_access_request | 비공개\s*공고 | true | 비공개 공고 허위 안내 차단 |
| citation_bypass | 출처(?:를)?\s*생략 | true | 출처 생략 지시 차단 |
| citation_bypass | 근거\s*없이 | false | 근거 없이 확정적 답변 차단 |
| citation_bypass | 출처\s*없이 | false | 출처 없이 답변 차단 |
| citation_bypass | 인용\s*없이 | false | 인용 없이 답변 차단 |

4가지 카테고리가 존재한다.

**prompt_injection**: LLM이 이전 시스템 지시를 무시하도록 유도하는 시도를 차단한다. 이 카테고리는 allowNegation이 false이므로, "이전 지시를 무시하지 않습니다"와 같은 표현도 차단된다. 프롬프트 인젝션에 대한 부정 표현 자체가 의미 없기 때문이다.

**guaranteed_outcome_claim**: 취업, 합격, 채용, 내정에 대한 보장 주장을 차단한다. 7개 패턴으로 구성되며, 모두 allowNegation이 true다. 즉, "취업을 보장하지 않습니다"와 같은 부정 표현은 허용된다. 이는 시스템이 스스로 보장을 부인하는 표현이므로 위험하지 않다.

**official_endorsement_claim**: 공식 인증, 승인, 학교 제휴, 보증, 공인 서비스에 대한 허위 주장을 차단한다. 5개 패턴이며 allowNegation이 true다. "공식 인증 서비스가 아닙니다"와 같은 표현은 안전하다고 판정한다.

**private_access_request**: SSO 인증이나 비공개 공고에 대한 접근 요구를 차단한다. 2개 패턴이며 allowNegation이 true다.

**citation_bypass**: 출처나 근거 없이 답변하려는 시도를 차단한다. 4개 패턴 중 3개("근거 없이", "출처 없이", "인용 없이")는 allowNegation이 false다. 이 세 표현은 그 자체로 위험한 행위를 지시하므로, 부정 형태로 표현되더라도 안전하다고 간주하지 않는다.

## 4. 마크다운/HTML 표면 탐지

LLM이 답변에 마크다운 링크나 HTML 태그를 삽입하면, 사용자가 해당 링크를 통해 신뢰할 수 없는 외부 페이지로 이동할 위험이 있다. output-safety-policy는 5가지 패턴으로 이러한 위험한 표면을 탐지한다.

### 탐지 패턴

```typescript
// 마크다운 이미지: ![alt](url)
const markdownImagePattern = /!\[[^\n]*?\]\([^\n)]*\)/gu;

// 마크다운 링크: [text](url) (이미지 패턴과 겹치지 않도록 부정 lookbehind 사용)
const markdownLinkPattern = /(?<!!)\[[^\n]+?\]\([^\n)]*\)/gu;

// 오토링크: <https://...> 또는 <mailto:...>
const autolinkPattern = /<(?:(?:https?:\/\/|mailto:)[^\s<>]+|[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,})>/gu;

// raw URL: https://... 또는 www...
const rawUrlPattern = /\b(?:https?:\/\/|www\.)[^\s<>()]+/giu;

// raw HTML 태그: <a>, <img>, <div> 등
const rawHtmlTagPattern = /<\/?[A-Za-z][A-Za-z0-9:-]*(?:\s[^<>]*)?>/gu;
```

### 차단 대상 표면 5가지

| 유형 | 예시 | 차단 사유 |
|------|------|-----------|
| 마크다운 링크 | `[공식 페이지](https://example.com)에서 보세요` | 사용자가 외부 링크 클릭 유도 |
| 오토링크 | `<https://example.com>에서 보세요` | 클릭 가능한 URL 표면 노출 |
| raw URL | `https://example.com/raw-url 에서 보세요` | 텍스트 형태의 외부 링크 노출 |
| 마크다운 이미지 | `![공고 이미지](https://example.com/image.png)` | 이미지 로딩을 통한 외부 리소스 요청 |
| HTML 태그 | `<a href="https://example.com">공고</a>` | 렌더링 가능한 HTML 요소 삽입 |

## 5. 숨겨진 인용 마커 탐지 로직

LLM이 인용 마커([1], [2] 등)를 답변에 포함하되, 이를 코드블록이나 인라인코드, 이미지 alt 텍스트, 링크 텍스트 안에 숨기면 사용자에게 가짜 인용이 노출되는 문제가 발생한다. output-safety-policy는 이를 방지하기 위해 **마스킹 기반 가시성 검사**를 수행한다.

### 검사 원리

```
원본 답변:  "자세한 내용은 [1]을 참고하세요. 코드: `[1]`"
                    ↑ 가시적            ↑ 비가시적

마스킹 후:  "자세한 내용은 [1]을 참고하세요. 코드: `XXX`"

추출 결과:  [1]  (코드블록 내부의 [1]은 제외됨)
```

### 마스킹 순서

```typescript
function maskNonVisibleCitationContexts(answer: string): string {
  const chars = [...answer.replace(/\r\n/gu, "\n").replace(/\r/gu, "\n")];

  maskFencedCode(chars);                          // 1. ```코드 블록``` 전체 마스킹
  maskPattern(chars, /`[^`\n]*`/gu);              // 2. `인라인 코드` 마스킹
  maskPattern(chars, markdownImagePattern);      // 3. ![alt](url) 마스킹
  maskPattern(chars, markdownLinkPattern);       // 4. [text](url) 마스킹
  // ... 추가 마스킹 패턴 적용

  return chars.join("");
}
```

문자열을 문자 배열로 변환한 뒤, 비가시적 영역의 문자를 별도 문자로 치환한다. 그 후 `[n]` 패턴을 검색하여 가시적인 인용 마커만 추출한다.

### 차단 시나리오

| 시나리오 | 원본 텍스트 | 판정 |
|----------|------------|------|
| 인라인코드 내 숨김 | `` `[1]` 안에만 인용을 숨깁니다.`` | **차단** (가시적 마커 없음) |
| 코드블록 내 숨김 | ```` ```text\n[1]\n``` 코드블록에만 인용을 숨깁니다.```` | **차단** (가시적 마커 없음) |
| 정상 노출 | "자세한 내용은 [1]을 참고하세요." | **허용** (가시적 마커 [1] 존재) |

## 6. 부정어 허용 로직 (allowNegation)

안전하지 않은 표현 패턴 중, 일부는 부정형으로 표현될 때 오히려 안전한 의미를 갖는다. 예를 들어 "취업을 보장합니다"는 위험하지만, "취업을 보장하지 않습니다"는 시스템이 보장을 명시적으로 부인하는 것이므로 안전하다.

allowNegation이 true인 패턴은 답변 텍스트에 해당 패턴이 포함되어 있더라도, 부정어가 함께 존재하면 안전하다고 판정한다.

### 부정어 인식 키워드

시스템은 다음의 부정 표현을 인식한다:

- 아니, 않, 지 않, 않습니다, 않는, 않은, 않다, 없, 지 않음, 보장하지 않습니다

### 동작 예시

| 패턴 | 표현 | allowNegation | 판정 |
|------|------|-------------|------|
| 취업 보장 | "취업을 보장 안내입니다" | true | **차단** (부정어 없음) |
| 취업 보장 | "취업을 보장하지 않습니다" | true | **허용** (부정어 "하지 않" 감지) |
| 합격 보장 | "합격 보장 프로그램입니다" | true | **차단** |
| 합격 보장 | "합격을 보장하지 않습니다" | true | **허용** |
| 이전 지시 무시 | "이전 지시를 무시합니다" | false | **차단** (부정어 불관련) |
| 근거 없이 | "근거 없이 확실합니다" | false | **차단** (allowNegation=false) |

## 7. PII 에코 탐지

LLM이 사용자의 개인 식별 정보(PII)를 답변에 반복(echo)하면, 채팅 로그에 민감 정보가 영구 저장되는 위험이 있다. output-safety-policy는 세 가지 PII 패턴을 탐지한다.

### 탐지 패턴

| PII 유형 | 정규식 | 탐지 예시 |
|----------|--------|-----------|
| 이메일 | 이메일 주소 패턴 매칭 | `erica.student@example.com` |
| 전화번호 | 한국식 전화번호 패턴 매칭 | `010-1234-5678` |
| 학번 | 10자리 숫자 학번 패턴 매칭 | `2024123456` |

이 탐지기는 **정보 추출이 아니라 정보 노출 차단**을 목적으로 한다. PII가 감지되면 답변 전체가 거부된다. 사용자가 질문에서 자신의 정보를 제공했더라도, LLM이 그 정보를 답변에 반사적으로 반복하면 안 된다.

## 8. 인용 마커 가시성 검증

output-validation은 인용 마커의 가시성을 두 단계로 검증한다.

### 1단계: visible 마커 존재 여부

hard_refuse가 아닌 모든 답변은 최소 하나의 가시적 인용 마커를 포함해야 한다. 가시적 마커가 없는 팩추얼 답변은 "출처 없는 주장"으로 간주하여 거부한다.

### 2단계: 마커 ↔ 객체 매핑

visible 마커의 ID가 `structuredCitations` 배열에 실제로 매핑되는지 검증한다. 답변에 [1], [3]이 포함되어 있지만 structuredCitations에 id 1과 3에 대한 객체가 없다면, 해당 답변은 거부된다.

### HTTPS 전용 URL 검증

모든 citation 객체의 url 필드가 `https://` 프로토콜로 시작하는지 검증한다. HTTP URL은 중간자 공격(man-in-the-middle)의 위험이 있으므로 허용하지 않는다.

### 캐노니컬 응답 검증

provider가 반환한 메타데이터를 기반으로 최종 캐노니컬 응답 객체를 생성하고, 이를 다시 Zod 스키마로 검증한다. 이중 파싱(double-parse)을 통해, 검증 로직이 놓친 경계 케이스를 Zod 스키마가 최종적으로 포착한다.

## 10. 실험 시나리오 상세

### 10.1 안전한 부정 표현 허용

시스템이 보장을 부인하거나, 비공개 접근을 거부하거나, 출처 생략을 지시하지 않는다는 표현은 안전하다.

| 입력 표현 | 판정 | 이유 |
|-----------|------|------|
| "취업을 보장하지 않습니다" | **허용** | guarantee 패턴 + 부정어 "하지 않" 감지 |
| "공식 인증 서비스가 아닙니다" | **허용** | official 패턴 + 부정어 "아닙니다" 감지 |
| "SSO로 확인하지 않습니다" | **허용** | private_access 패턴 + 부정어 감지 |
| "출처를 생략하지 않습니다" | **허용** | citation_bypass 패턴 + 부정어 감지 |

부정 표현이 허용될 때, 해당 답변에 가시적 인용 마커와 유효한 인용 티어가 함께 포함되어야 한다는 점에 유의한다. 부정 표현 허용은 구문 안전성 패스를 의미할 뿐, 다른 검증 항목(인용 가시성, HTTPS URL 등)의 통과를 보장하지 않는다.

### 10.2 불안전한 보장 표현 차단 (7가지 변형)

| # | 입력 표현 | 패턴 | 판정 |
|---|-----------|------|------|
| 1 | 취업 보장 안내입니다 [1] | 취업 보장 | **차단** |
| 2 | 합격 보장 프로그램입니다 [1] | 합격 보장 | **차단** |
| 3 | 채용 보장 과정입니다 [1] | 채용 보장 | **차단** |
| 4 | 합격 확정 대상입니다 [1] | 합격 확정 | **차단** |
| 5 | 100% 채용 연계입니다 [1] | 100% 채용 | **차단** |
| 6 | 내정 보장 혜택입니다 [1] | 내정 보장 | **차단** |
| 7 | 프리패스 전형입니다 [1] | 프리패스 | **차단** |

모든 변형에 visible citation marker [1]이 포함되어 있지만, 안전성 패턴이 먼저 위반되므로 차단된다. 인용이 있더라도 보장 주장 자체가 허용되지 않는다.

### 10.4 PII 에코 차단 (3가지 유형)

| 유형 | 예시 입력 | 판정 |
|------|-----------|------|
| 이메일 | 학생 이메일은 erica.student@example.com 입니다 | **차단** |
| 전화번호 | 연락처는 010-1234-5678 입니다 | **차단** |
| 학번 | 학번은 2024123456 입니다 | **차단** |

LLM이 사용자 질문에 포함된 개인정보를 답변에 반복하는 것을 방지한다. 이 검사는 답변 전체 텍스트를 대상으로 수행되며, 인용 마커나 다른 안전성 검사와 무관하게 PII가 감지되면 즉시 차단된다.

### 10.5 마크다운/HTML 표면 차단 (5가지)

| 유형 | 예시 입력 | 판정 |
|------|-----------|------|
| 마크다운 링크 | `[공식 페이지](https://example.com)에서 보세요` | **차단** |
| 오토링크 | `<https://example.com>에서 보세요` | **차단** |
| raw URL | `https://example.com/raw-url 에서 보세요` | **차단** |
| 마크다운 이미지 | `![공고 이미지](https://example.com/image.png) 확인하세요` | **차단** |
| HTML 태그 | `<a href="https://example.com">공고</a> 확인하세요` | **차단** |

LLM이 생성하는 모든 외부 링크 참조는 차단된다. 출처 정보는 structuredCitations 객체를 통해 프론트엔드의 전용 출처 카드 UI로 렌더링되어야 하며, LLM 답변 텍스트 내에 직접 링크가 포함되어서는 안 된다.

### 10.6 숨겨진 인용 마커 차단 (2가지)

| 유형 | 예시 입력 | 판정 | 원인 |
|------|-----------|------|------|
| 인라인코드 | `` `[1]` 안에만 인용을 숨깁니다.`` | **차단** | 비가시적 영역 내부에만 마커 존재 |
| 코드블록 | ```` ```text\n[1]\n``` 코드블록에만 인용을 숨깁니다.```` | **차단** | 비가시적 영역 내부에만 마커 존재 |

이 검사는 마스킹 로직을 통해 비가시적 영역(코드블록, 인라인코드, 이미지, 링크)을 제거한 후 남은 가시적 영역에서만 인용 마커를 카운트한다. 가시적 영역에 마커가 없으면 답변이 인용 없는 것으로 판정되어 거부된다.

## 11. 결론

Layer 5 출력 검증은 64개의 테스트를 모두 통과하며, 다음 네 가지 보장을 입증했다.

**첫째, 허위 보장 주장 차단.** 취업, 합격, 채용, 내정에 대한 7가지 보장 패턴을 탐지하여, LLM이 허위 보장을 생성하는 것을 방지한다. 부정어 허용 로직을 통해 "보장하지 않습니다"와 같은 안전한 부인 표현은 정상적으로 허용된다.

**둘째, 공식 인증 위조 차단.** 공식 인증, 승인, 한양대 제휴, 학교 보증, SSO 확인, 비공개 공고 등 7가지 패턴으로, LLM이 존재하지 않는 공식 관계를 주장하는 것을 차단한다. 이는 AGENTS.md에 명시된 "한양대 보증을 주장하지 않는다"는 프로젝트 규칙의 직접적인 구현이다.

**셋째, 외부 링크 노출 차단.** 마크다운 링크, 오토링크, raw URL, 마크다운 이미지, HTML 태그 등 5가지 표면 패턴으로, LLM 답변 내의 모든 외부 링크 참조를 차단한다. 출처 정보는 프론트엔드 UI의 전용 출처 카드를 통해서만 제공되어야 한다.

**넷째, 인용 무결성 보장.** 숨겨진 인용 마커 탐지, 마커-객체 매핑 검증, HTTPS 전용 URL 검증을 통해, 인용 시스템의 무결성을 보장한다. LLM이 인용을 코드블록이나 인라인코드에 숨기거나, 존재하지 않는 인용 ID를 참조하거나, 비보안 URL을 인용하는 것을 모두 차단한다.

이 네 가지 보장은 fail-closed 설계 원칙에 따라 동작한다. 하나의 위험 신호라도 감지되면 답변 전체가 거부된다. 이는 "안전하지 않은 출력은 절대 사용자에게 노출하지 않는다"는 Layer 5의 핵심 철학을 충실히 구현한 결과다.
